# 汽车配件仓库管理系统 — AI 智能助手技术说明

## 一、它做什么

用户用自然语言跟系统对话，比如：
- "前车门有多少" → 查库存 → 返回 "前车门当前库存 21 个，安全库存 3 个"
- "火花塞还有多少" → 查库存 → 返回 "火花塞当前库存 0 支，安全库存 5 支 ⚠️低于安全库存"
- "今天入库了多少件" → 统计 → 返回 "今天入库单数量：0 单"
- "怎么新增供应商" → 查知识库 → 返回 4 步操作指南
- "打开入库页面" → 页面导航 → 前端跳转到入库管理页面

不用记菜单位置、不用学查询语法，说人话就行。

## 二、用到的技术

| 层 | 技术 | 干什么 |
|---|------|--------|
| 前端 | Vue 2.7 + Element UI | 右下角悬浮对话面板，SSE 流式接收 |
| 后端 | Spring Boot 2.7 + MyBatis-Plus | AI 服务 + 业务接口调用 |
| AI 模型 | 阿里云百炼 Qwen3-Max | 理解用户意图、生成回复 |
| AI 框架 | LangChain4j 1.0.0 | 封装 LLM 调用、向量嵌入 |
| 向量模型 | text-embedding-v3 | 把文档转成向量 |
| 向量存储 | InMemoryEmbeddingStore | 存向量、做相似度搜索 |
| 通信方式 | SSE (Server-Sent Events) | 流式推送回复，打字机效果 |

## 三、核心架构：Prompt-Based Function Calling

### 为什么不用原生 Function Calling

阿里云百炼 DashScope 的 OpenAI 兼容接口不支持原生的 `tool_calls` 协议。
所以用了"提示词描述工具 + LLM 返回 JSON 调用指令"的方案。

### 流程（一次对话）

```
用户输入 "火花塞还有多少"
        │
        ▼
┌─────────────────────────────────┐
│  AIService.processMessage()     │
│                                 │
│  构建 System Prompt：           │
│  "你是仓库管理助手..."          │
│  + 8 个可用工具定义             │
│  + JSON 调用格式说明            │
│  + 示例：{"tool":"queryStock... │
│        │                        │
│        ▼                        │
│  LLMClient.chatWithSystem()     │
│  → DashScope API                │
│  → qwen3-max 模型               │
│        │                        │
│        ▼                        │
│  LLM 返回 JSON：                │
│  {"tool":"queryStockByPart",    │
│   "params":{"partName":"火花塞"}}│
│        │                        │
│        ▼                        │
│  extractToolCall() 解析 JSON    │
│        │                        │
│        ▼                        │
│  WarehouseTools.dispatch()      │
│  → queryStockByPart("火花塞")   │
│  → partService.listByName()     │
│  → stockService.getByPartId()   │
│        │                        │
│        ▼                        │
│ 返回: "【火花塞】当前库存：0支" │
└─────────────────────────────────┘
```

### 8 个可用工具

| 工具名 | 做什么 | 调用的业务接口 |
|--------|--------|---------------|
| queryStockByPart | 按名称查配件库存 | partService.listByName() + stockService.getByPartId() |
| queryLowStock | 查低于安全库存的配件 | stockService.getLowStockList() |
| navigateToPage | 返回页面路由路径 | 内置路由映射表 |
| getStatistics | 今日/本月入库出库统计 | inboundService.getTodayCount() 等 |
| queryPartCount | 配件种类数 | partService.count() |
| queryCategoryCount | 分类数 | categoryService.count() |
| queryPendingApproval | 待审批出库单数 | outboundService.countPendingApproval() |
| searchKnowledgeBase | RAG 知识库检索 | RAGService.query() → 向量搜索 → LLM 生成 |

### 和旧方案的区别

旧方案：120 行 if-else 关键词匹配 + 正则提取配件名。
问题：用户说"火花塞还有多少"→ 正则把"还有多少"去掉但漏了"还"→ 配件名变成"火花塞还"→ 查不到。

新方案：LLM 理解整句话的语义，自己决定调哪个工具、传什么参数。
"火花塞还有多少"和"查火花塞库存"都会被路由到 queryStockByPart("火花塞")，
不需要写死关键词。

## 四、RAG 知识库是怎么做的

### 不是全文注入

旧方案把整个操作手册复制粘贴到 Prompt 里发给 LLM，内容越多越慢越贵。

### 现在用向量检索

```
系统启动时
  │
  ├─ 加载 knowledge-base/ 下 5 个 MD 文档
  ├─ DocumentByParagraphSplitter(500字, 50字重叠) 切成 13 个段落
  ├─ text-embedding-v3 把每个段落转成向量（每批 8 个，DashScope 限制 ≤10）
  └─ 存入 InMemoryEmbeddingStore

用户问 "怎么新增供应商"
  │
  ├─ 查询语句 → text-embedding-v3 → 查询向量
  ├─ embeddingStore.search(查询向量, TopK=3, minScore=0.3)
  ├─ 找到最相关的 3 个段落
  ├─ 拼接成 Prompt："参考以下知识库内容回答：...[3个段落]... 用户问题：怎么新增供应商"
  └─ LLM 生成回答
```

### 关键参数
- 分块大小：500 字符，重叠 50 字符
- 检索数量：Top 3
- 相似度阈值：0.3（太低会返回不相关内容）
- 嵌入模型：text-embedding-v3
- 嵌入批次：每批 8 个（DashScope API 限制单次 ≤10）

## 五、SSE 流式输出怎么实现

```
前端发起 GET /api/ai/chat/stream?message=xxx
  │
  ├─ 后端先走 AIService (Prompt-Based Function Calling)
  │   └─ LLM + 工具调用 → 拿到完整回复文本
  │
  ├─ 后端把文本切成小块（中文 2-4 字/块，标点单独切）
  │
  ├─ 每 20-50ms 发一个 SSE token 事件
  │   event:token
  │   data:【火花
  │   event:token
  │   data:塞】当
  │   ...
  │
  └─ 最后发 done 事件
      event:done
      data:{"historyCount":2}
```

前端用 `EventSource` API 接收，收到 token 就追加到消息气泡里，看起来像打字。

## 六、后端接口一览

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/ai/chat` | POST | 非流式对话，返回完整结果 |
| `/api/ai/chat/stream` | GET | SSE 流式对话，打字机效果 |
| `/api/ai/status` | GET | AI 服务状态（是否启用、模式、调用统计） |
| `/api/ai/stats` | GET | 详细统计（审计日志、限流状态、最近记录） |

### 速率限制
- 每个会话每分钟最多 20 次请求
- 超过返回 429 + "请求过于频繁"
- 滑动窗口算法，自动清理过期窗口

### 审计日志
- 记录每次调用：时间、会话ID、用户输入、AI回复、token估算、耗时
- 内存保留最近 500 条
- 可查最近 10 条

## 八、面试问答参考

**Q: 这个 AI 助手怎么做的？**

A: 核心思路是用大模型做意图识别，然后调用真实的业务接口获取数据。

具体做法：后端构建了一个 System Prompt，里面描述了 8 个可用"工具"——比如查库存、查统计、搜知识库等，每个工具都有名字、参数和调用格式。用户发来问题后，Prompt + 用户问题一起发给阿里云的 Qwen3-Max 模型。模型理解语义后返回一个 JSON，比如 `{"tool":"queryStockByPart","params":{"partName":"火花塞"}}`。后端解析这个 JSON，调用对应的业务方法去数据库查数据，把结果返回给用户。

这样做的好处是：LLM 只负责理解用户想干什么，真正的数据查询走的还是原来的业务接口，数据不会出错。而且用户不管说"火花塞还有多少"还是"查火花塞库存"，LLM 都能正确理解成同一个操作，不像以前用关键词匹配那样容易搞错。

**Q: RAG 是怎么回事？**

A: RAG 就是"检索增强生成"。简单说就是先把系统的操作手册（怎么新增供应商、怎么审批出库单等）预处理掉——切成段落、用嵌入模型转成向量存起来。用户问操作问题的时候，先把问题也转成向量，去向量库里搜最相关的几个段落，把这些段落作为"参考资料"和用户问题一起发给大模型。这样大模型就能基于真实的操作手册来回答，不会瞎编。

具体用的是 LangChain4j 框架 + 阿里云的 text-embedding-v3 嵌入模型，分块大小 500 字，每次检索返回 Top 3 最相关的段落。

**Q: 流式输出怎么实现？**

A: 后端先完成工具调用拿到完整回复，然后把文本切成小块（中文 2-4 个字一块），通过 SSE（Server-Sent Events）每隔几十毫秒推一个 `token` 事件给前端。前端用 `EventSource` API 接收，逐步追加到消息气泡里，看起来像 ChatGPT 那种打字效果。

**Q: 数据安全怎么保证？**

A: 第一，所有数据查询都调用现有的 Service 层业务方法，LLM 不直接接触数据库。第二，AI 只能查数据、返回操作指引，不能执行新增/删除/修改操作。第三，有速率限制，每个会话每分钟最多 20 次请求。第四，所有 LLM 调用都有审计日志记录。

**Q: 为什么用 Prompt 描述工具而不是真正的 Function Calling？**

A: 阿里云百炼的 DashScope 接口虽然兼容 OpenAI 格式，但不支持原生的 `tool_calls` 字段。所以变通了一下——把工具定义写在 System Prompt 里，让模型返回 JSON 格式的调用指令，后端再解析执行。效果一样，只是实现方式不同。如果以后 DashScope 支持原生 Function Calling 了，迁移也很简单。

**Q: 用了哪些技术栈？**

A: 前端 Vue 2.7 + Element UI，后端 Spring Boot 2.7 + MyBatis-Plus，AI 模型用的是阿里云百炼的 Qwen3-Max（对话）和 text-embedding-v3（向量嵌入），AI 框架用了 LangChain4j 1.0.0（封装 LLM 调用和向量检索），通信是 SSE 流式 + REST API。

## 九、简历描述参考
