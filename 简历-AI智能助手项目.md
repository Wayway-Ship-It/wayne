# 项目经历

## 汽车配件仓库管理系统 — AI 智能助手

**项目时间**：2026.01 - 2026.06
**担任角色**：全栈开发（独立完成）
**项目地址**：私有仓库

---

### 项目背景

传统仓库管理系统操作门槛高，查询库存要记住配件名称、找到对应菜单、手动筛选。新员工培训周期长，日常查询效率低。目标是让用户用自然语言就能操作系统——说"前车门有多少"直接看到库存，说"怎么新增供应商"直接拿到操作指南。

### 技术栈

**后端**：Spring Boot 2.7、MyBatis-Plus、MySQL
**前端**：Vue 2.7、Element UI
**AI 引擎**：阿里云百炼 Qwen3-Max（对话）、text-embedding-v3（向量嵌入）
**AI 框架**：LangChain4j 1.0.0
**通信**：REST API、SSE 流式推送

### 我的工作

**1. 设计结构化工具调度架构（Prompt 驱动的 JSON 生成）**
← 🎤 "说白了就是：写一段角色设定+工具清单给大模型，大模型读完用户的问题后，自己决定要不要调工具、调哪个，然后返回一个 JSON 告诉我它的决定，我再按 JSON 的指示去执行。"

阿里云百炼的 DashScope 接口兼容 OpenAI 格式但不支持原生 `tool_calls`，因此采用 Prompt 描述工具 + LLM 返回 JSON 调用指令的方案（功能上等价于 Function Calling）：
← 🎤 "OpenAI 的 Function Calling 是模型原生输出一个 function_call 字段，阿里的接口不支持这个字段，所以我换了个思路——用纯文本 Prompt 告诉模型'你可以用这些工具'，让模型用 JSON 格式告诉我它想调哪个。"

- 在 System Prompt 里定义了 8 个结构化"工具"（查询配件库存、低库存预警、统计数据、知识库检索、页面导航等），每个工具有名字、参数说明、JSON 调用示例
← 🎤 "System Prompt 就是每次对话前先塞给模型的'角色设定+规则说明书'，告诉它你是谁、能干什么、回答格式是什么。"
- 用户消息发给 Qwen3-Max 后，模型判断是否需要调用工具，需要就返回 `{"tool":"queryStockByPart","params":{"partName":"火花塞"}}`
← 🎤 "这个 JSON 就是模型'思考'后的输出——它理解了用户想问库存，自己选好了工具名和参数。这个 JSON 不是我们手动拼的，是模型根据 Prompt 里的示例格式生成的，所以参数名和结构都是可控的。"
- 后端写了 `extractToolCall()` 解析 JSON → `dispatch()` 方法用 switch 分发到对应工具 → 工具内部调用 Service 层查数据库
← 🎤 "收到模型的 JSON 后，后端做三件事：① 用 Jackson 解析 JSON 拿到工具名和参数；② 用一个 switch 根据工具名路由到对应的处理方法；③ 方法里调已有的 Service 层代码去查数据库。模型只管'意图理解'，数据的真实来源还是业务数据库。"
- 这套架构替代了之前的 120 行 if-else 关键词匹配，解决了"火花塞还有多少"被正则错误提取为"火花塞还"的问题
← 🎤 "旧方案靠正则表达式去掉'还有多少'这类口语词来提取配件名，但正则总有漏掉的边界情况。LLM 方案从语义层面理解整句话，不管用户换什么说法都能正确提取。"

关键代码结构：
```
AIService（主引擎）
  ├─ SYSTEM_PROMPT（角色设定 + 工具列表）
  ├─ 调用 LLMClient → DashScope API → Qwen3-Max
  ├─ 解析返回 JSON → extractToolCall()
  └─ WarehouseTools.dispatch() → 8 个工具方法
```
← 🎤 "整体流程：System Prompt + 用户消息 → 发给模型 → 模型返回 JSON → 解析 → 调度到具体工具 → 返回真实数据。模型只做'翻译官'，不做'数据库'。"

**2. 实现 RAG 向量检索**

旧方案是把整个操作手册文本直接拼进 Prompt 发给 LLM，文档越长越慢；改成了真正的向量检索：
← 🎤 "RAG 的核心思想：不问问题时不带全书，问了问题只找最相关的几页。先把知识库文档切成小块，转成数学向量存起来；用户提问时，把问题也转成向量，用余弦相似度找到最相关的片段，只把这些片段和问题一起发给模型。"

- 系统启动时加载 5 份知识库 MD 文档（操作手册、供应商管理、库存管理、审批流程、仓库日常），按 500 字符固定长度切分（重叠 50 字符，保证段落边界上下文不丢失），共生成 13 个文本块
← 🎤 "500 字符一段、重叠 50 字符——这个参数是试出来的。太短了语义不完整，太长了检索精度下降。重叠是为了防止正好在段落边界切断关键信息。"
- 调阿里云 `text-embedding-v3` 接口把段落转成向量，每批 8 个（API 限制单次 ≤10）
← 🎤 "嵌入（Embedding）就是把一段文本变成一个高维数学向量，语义相近的文本向量也相近。阿里云的嵌入 API 一次最多传 10 条，我每批发 8 条留点余量。"
- 存入 `InMemoryEmbeddingStore`
← 🎤 "存到一个内存向量库里，方便按相似度快速检索。目前是内存方案，后续可以换 Chroma 等持久化向量库。"
- 用户提问时先转向量 → 余弦相似度搜 Top 3（minScore=0.3）→ 拼成增强 Prompt → 发给 LLM 生成回答
← 🎤 "检索流程：① 把用户问题也转成向量；② 在向量库里用余弦相似度找最接近的 3 个文档片段（低于 0.3 分的不要，说明不相关）；③ 把这 3 段作为'参考资料'和用户问题一起拼到 Prompt 里；④ 模型基于参考资料生成答案，而不是凭空编。"

**3. 实现 SSE 流式推送**

- 后端完成工具调用与 LLM 生成后，将回复文本按自适应规则分块（中文 2-4 字/块、标点单独断句），通过 SSE 每 20-50ms 推送一个 token 事件，最后发 done 事件
← 🎤 "SSE 是 HTTP 长连接，服务器可以持续向浏览器推送数据，比 WebSocket 轻量。前端不需要引入额外的库，浏览器原生 EventSource API 就能接。分块策略：中文按 2-4 个字切、标点符号单独成块，配合 20-50ms 的间隔，视觉上就像在打字。"
- 前端用 `EventSource` API 逐步接收并追加到消息气泡，实现打字机效果

**4. 辅助功能**

- 滑动窗口速率限制：每个会话每分钟最多 20 次，超过返回 429，用 `ConcurrentHashMap` 存窗口计数器
← 🎤 "限流是防止用户或脚本刷接口。滑动窗口比固定窗口更平滑——比如每分钟 20 次，固定窗口在 12:00:59 和 12:01:00 各发 20 次就不拦，滑动窗口会拦。"
- 审计日志：内存缓冲 500 条，记录每次调用的时间、会话、输入输出、token 数、耗时
← 🎤 "审计日志是为了可追溯——每次 LLM 调用记了谁、问了什么、答了什么、花了多少 token、耗时多久，方便排查问题和统计用量。"
- 多轮对话记忆：`InMemoryChatMemoryStore` 存每会话最多 20 条消息，对话历史拼入 Prompt 实现上下文理解
← 🎤 "没有对话记忆的话，每次问都是'全新对话'，模型不知道你上一句说了什么。把最近 20 条对话历史拼进 Prompt，模型就能上下文连续对话了。"

### 遇到并解决的问题

- **DashScope 不支持原生 Function Calling**：改用 Prompt 描述工具 + JSON 返回的方案，效果等价
- **DashScope 嵌入 API 单次限制 ≤10 条**：`EmbeddingStoreIngestor` 默认一次全发，改为手动分批每批 8 条
← 🎤 "框架默认行为是一次把所有文档段落发给嵌入 API，但阿里限制单次最多 10 条。不处理的话直接报错。改成手动循环每 8 条一批解决。"
- **关键词匹配的边界情况**：用户说"火花塞还有多少"，正则 `replaceAll("还有多少|有几个", "")` 漏掉了"还"字 → 改用 LLM 语义理解从源头解决
- **Git Bash 中文编码**：curl 传中文 JSON 乱码，改用 `echo -n > file` + `-d @file` 方式

### 项目成果

- 意图识别与路由模块从原有的 120 行关键词匹配 + 意图解析，精简为 1 套 Prompt 配置 + 80 行通用调度逻辑，新增意图只需修改 Prompt 配置，无需改动调度代码
- 日常查询（查库存、统计、导航）LLM 一次调用完成，平均耗时 800ms-2s
- 知识库操作指引通过 RAG 向量检索 + LLM 生成，答案基于真实文档
- 速率限制 + 审计日志支撑生产使用
